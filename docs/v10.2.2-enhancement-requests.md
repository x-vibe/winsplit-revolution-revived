## Bug Reports

### BUG-001: Portable mode does not save settings next to executable

**Severity:** Medium
**Component:** Settings / XML Configuration
**Discovered:** 2026-02-27

**Description:**
When running WinSplit Revolution Revived v10.2.0 in portable mode (with `portable.txt` present in the application directory), settings are **not** saved next to the executable as expected. Instead, they are written to:

```
%APPDATA%\Winsplit Revolution\
```

Specifically, the following files are created in AppData rather than the portable directory:
- `hotkeys.xml` (custom keyboard shortcuts)
- `layout.xml` (layout/window positioning settings)

**Expected behavior:**
When `portable.txt` exists in the same directory as `Winsplit.exe`, all configuration files (`hotkeys.xml`, `layout.xml`, and any other settings) should be saved in the application directory, making the installation fully self-contained and portable.

**Actual behavior:**
The `portable.txt` marker file is ignored and settings are always written to `%APPDATA%\Winsplit Revolution\`.

**Steps to reproduce:**
1. Extract the v10.2.0 portable ZIP to any folder
2. Verify `portable.txt` exists next to `Winsplit.exe`
3. Launch `Winsplit.exe`
4. Change any hotkey or layout setting
5. Close WinSplit
6. Check the application directory — no XML files are created
7. Check `%APPDATA%\Winsplit Revolution\` — `hotkeys.xml` and `layout.xml` are found there

**Likely cause:**
The portable path detection logic in the settings save/load code may not be checking for `portable.txt` correctly, or the wxWidgets 3.x migration may have changed how the application directory is resolved at runtime.

## Enhancement Requests

### ENH-001: Modernize dialog windows with Windows 11 design language

**Priority:** Medium
**Component:** UI / Dialogs
**Requested:** 2026-02-27

**Description:**
All popup windows and dialog boxes in WinSplit Revolution currently use legacy Windows visual styles that look outdated on Windows 10 and Windows 11. This includes the Options dialog (all tabs), the Hotkey Settings dialog, the Layout Settings dialog, the About dialog, the Active Window Tools dialog, and any confirmation/warning message boxes.

**Current state:**
- Dialogs use classic Windows control styles (flat buttons, old-style tabs, legacy fonts)
- Message boxes use the basic Win32 MessageBox appearance
- No visual consistency with the modern Windows 11 UI (rounded corners, Mica/Acrylic materials, Segoe UI Variable font, WinUI 3 control styles)

**Requested changes:**
1. **Enable Windows 11 visual styles** — Ensure the application manifest and wxWidgets initialization properly activate modern common controls (ComCtl32 v6) for all dialogs
2. **Use modern dialog layouts** — Update dialog designs to follow Windows 11 UX guidelines (increased padding, modern typography with Segoe UI Variable, cleaner grouping)
3. **Rounded corners and modern controls** — Leverage wxWidgets 3.x theming support for native Windows 11 control rendering where possible
4. **Dark mode support** — Respect the system dark/light mode preference for all dialogs and popup windows
5. **Modern message boxes** — Use TaskDialog (Vista+) instead of legacy MessageBox for richer, more informative notifications with icons, expandable details, and command links where appropriate
6. **Consistent icon style** — Use the Fluent UI System Icons (already partially integrated per v10.2.0 changelog) across all dialog windows, not just menus

**Reference:**
- [Windows 11 Design Principles](https://learn.microsoft.com/en-us/windows/apps/design/)
- [Fluent UI System Icons](https://github.com/microsoft/fluentui-system-icons)
- wxWidgets 3.x supports native Windows theme rendering when manifested correctly
